# Axiom backend migration (halo2-ecc + halo2-base) ‚Äî production-ready on a 32 GB machine

> **Goal:** migrate ZKMist's entire circuit stack from the PSE halo2 backend to
> the **axiom** halo2 stack (`halo2-axiom` + `halo2curves-axiom` + `halo2-base`
> + `halo2-ecc`). This is the one path that delivers all three production wins
> at once: **k=18 (~1 GiB proving ‚Äî fits any machine, including 32 GB and
> claimants' laptops)**, an **audited secp256k1** (halo2-ecc), and a **wirable
> `keccak(pubkey)‚Üíaddress` bridge** (halo2-ecc exposes pubkey bytes ‚Äî the bridge
> halo2wrong's CRT limbs cannot do).
>
> Originates from the 2026-07-03 investigation: both PSE-compatible options are
> blocked (halo2wrong: no byte extraction; halo2-ecc-as-chip-swap: axiom curve
> conflict). The conflict is only avoided by migrating the **whole** backend.

## 1. Why this is worth a multi-week rewrite

| Path | Proving RAM | secp audit | Keccak bridge | Claimant UX |
|---|---|---|---|---|
| Audit hand-rolled secp (stay PSE, k=23) | ~25 GiB | via audit | works (byte-aligned limbs) | poor (heavy prove) |
| **Axiom migration (k=18)** | **~1 GiB** | **audited lib** | **works** | **great** |

For an airdrop targeting ~1M ordinary claimants, ~25 GiB `zkmist prove` excludes
most users; ~1 GiB lets anyone prove on a laptop. That tips the effort toward
worth-it.

## 2. The axiom stack (replaces PSE)

| PSE (current) | Axiom (target) | Role |
|---|---|---|
| `halo2_proofs` (PSE fork v0.3.0) | `halo2-axiom` (= `halo2_proofs_axiom`) | proving backend |
| `halo2curves` 0.6.1 | `halo2curves-axiom` 0.7.2 | curve types (`Fr`, `G1Affine`, ‚Ä¶) |
| ‚Äî (raw `Circuit`/`Layouter`) | `halo2-base` | `Context`/`RangeChip` eDSL + **Poseidon chip** |
| ‚Äî | `halo2-ecc` | **audited secp256k1** (`EccChip`, byte-extractable) |
| `halo2-solidity-verifier` (PSE) | axiom Solidity verifier (locate in `halo2-axiom`/`halo2-lib`) | on-chain `Halo2Verifier.sol` |

The two stacks **cannot coexist** in one circuit (different `Fr` types), so the
migration is all-or-nothing per circuit ‚Äî rewrite to the axiom eDSL.

## 4. The address binding becomes wirable (the halo2wrong blocker, solved)

halo2-ecc exposes the pubkey as bytes (`pk_bytes_le`, per PSE `zkevm-circuits`
`sig_circuit.rs`), so the `keccak(pubkey)‚Üíaddress` binding ‚Äî impossible with
halo2wrong's CRT limbs ‚Äî is just "feed halo2-ecc's pubkey bytes into the Keccak
input," exactly as PSE's zkEVM does it.

## 5. Verifier regeneration

The on-chain `Halo2Verifier.sol` / `Halo2VerifyingKey.sol` are currently
generated by PSE's `halo2-solidity-verifier` for the PSE transcript. After the
migration, regenerate with **axiom's Solidity verifier** (locate the exact tool
‚Äî likely in `halo2-axiom` or `halo2-lib`; phase-1 task). The verifier contract
format differs from PSE's; `ZKMAirdrop.claim`'s call signature must be
re-checked against the new verifier. The contract logic (nullifier map, cap,
mint) is unchanged.

## 7. Risks

- **API learning curve:** axiom's `Context`/`RangeChip` eDSL differs from PSE's
  `Layouter`/`Circuit`. (Mitigation: it's higher-level, often shorter; mirror
  halo2-ecc's own tests + PSE `zkevm-circuits` `sig_circuit` as references.)
- **Solidity verifier:** must locate/verify axiom's on-chain verifier matches
  the prover's transcript. (Phase-1 gate.)
- **Keccak port effort:** the hand-rolled Keccak is 1250 lines; porting to the
  eDSL is real work, OR adopt PSE `zkevm-circuits`'s axiom-stack Keccak (extra
  dep, but audited).
- **Multi-week scope:** this is a full circuit-layer rewrite, not a quick swap.

## 8. Effort

~2‚Äì4 weeks focused engineering, then audit of the (much smaller) integration +
Keccak port. Ends at: audited secp+Poseidon, k=18, ~1 GiB proving, byte-bridge
sound ‚Äî production-ready on a 32 GB machine (and claimants' laptops).

## 9. Phase 1 ‚Äî Poseidon port (DONE 2026-07-04)

Replaced the hand-rolled Poseidon permutation with the audited
`halo2_base::poseidon` chip, configured with the project's Circom parameters.
Deliverables:

- **`circuits/src/poseidon_axiom.rs`** ‚Äî axiom-stack Poseidon gadget exposing
  `hash_leaf` (t=2, RATE=1, R_F=8, R_P=56) and `hash_interior` (t=3, RATE=2,
  R_F=8, R_P=57), built on `halo2_base::poseidon::PoseidonHasher`. Coexists
  with the PSE `poseidon.rs` until the circuit is rewritten in Phase 3.
- **`circuits/tests/poseidon_axiom.rs`** ‚Äî three independent correctness proofs:
  1. **Parameters are Circom-compatible:** the raw Grain-LFSR permutation from
     `OptimizedPoseidonSpec` reproduces `light_poseidon::Poseidon::hash`
     byte-for-byte for both arities (strongest possible "match circom params"
     check).
  2. **`PoseidonChip::new(ctx, spec, range)` runs:** instantiating it loads all
     constants in-circuit and MockProver asserts every constraint.
  3. **In-circuit gadget == native sponge:** the chip's constrained output
     equals a native reference under halo2-base's sponge convention. Combined
     with (1), this confirms the chip uses the Circom permutation *and*
     implements halo2-base's sponge correctly.

All three pass; the PSE stack and `axiom_stack_foundation` test are unaffected.

### 9.1 Sponge-convention divergence ‚Äî Phase 3 reconciliation required

The PERMUTATION matches light-poseidon/Circom exactly (verified above), but the
HASH functions differ by construction:

| Aspect           | halo2-base (this gadget)               | light-poseidon / Circom (`zkmist-merkle-tree`) |
|------------------|----------------------------------------|------------------------------------------------|
| Capacity element | `2^64`                                 | `0`                                            |
| Input padding    | `1` then `0`s                          | none for a full RATE block                     |
| Squeeze          | extra empty permutation when `len % RATE == 0` | single permutation                      |
| Digest output    | `state[1]`                             | `state[0]`                                     |

The off-chain Merkle tree (`zkmist-merkle-tree`) and the existing PSE circuit
commit to the **light-poseidon** convention. Phase 3 must reconcile this so the
circuit verifies the *same* tree that is committed on-chain ‚Äî either:

- **(a)** rebuild the off-chain Merkle tree + nullifier under halo2-base's
  convention (audited, standard sponge ‚Äî preferred), or
- **(b)** wrap the axiom chip to replicate the Circom convention (capacity 0,
  no squeeze, output `state[0]`) so existing tree tooling keeps working.

This does not affect Phase 1/2 ‚Äî the chip and its parameters are correct either
way. It is a wiring decision for the full-circuit rewrite (Phase 3).

## 10. Phase 2 ‚Äî secp256k1 `scalar¬∑G` + pubkey byte-bridge (DONE 2026-07-04)

Proved, in an isolated circuit, the capability halo2wrong could NOT provide and
that motivated this whole migration: deriving the secp256k1 **public key bytes**
from `privkey ¬∑ G` so they can feed `keccak256 ‚Üí Ethereum address`. Deliverables:

- **`circuits/src/secp_axiom.rs`** ‚Äî axiom-stack secp256k1 gadget exposing:
  - `pubkey_from_privkey` ‚Äî `privkey ¬∑ G` via halo2-ecc's audited
    `EccChip::fixed_base_scalar_mult`.
  - `field_point_to_le_bytes` ‚Äî the **byte-bridge**: extract an secp256k1-Fp
    coordinate as 32 constrained little-endian bytes, ready for the Keccak
    preimage. (halo2-ecc's `ProperCrtUint` keeps a **positional** truncation
    alongside its CRT native cell ‚Äî unlike halo2wrong's pure CRT residues ‚Äî so
    each limb decomposes cleanly into bits/bytes.)
- **`circuits/tests/secp_axiom.rs`** ‚Äî two tests:
  1. `test_halo2ecc_secp256k1_pubkey_byte_bridge` ‚Äî in-circuit `scalar¬∑G` is
     constrained equal to the native pubkey (`FpChip::assert_equal`); the pubkey
     coordinates are extracted as 32 LE bytes each; the 64 bytes recompose to the
     coordinates exactly AND `keccak256(x_be‚Äñy_be)[12..]` equals the Ethereum
     address. MockProver (k=18, ~131k advice cells) asserts every constraint.
  2. `test_eth_address_known_vector_privkey_one` ‚Äî anchors the native
     `privkey¬∑G ‚Üí keccak(pubkey)` derivation to the canonical vector
     `privkey=1 ‚Üí 0x7E5F‚Ä¶5Bdf`, so the in-circuit-vs-native comparison is tied to
     Ethereum ground truth (not merely self-consistent).

### 10.1 Findings (don't re-discover)

- **Scalar chunking (handoff correction).** The handoff's `fixed_base_scalar_mult(
  ‚Ä¶, 256, 4)` is wrong: halo2-ecc asserts `max_bits ‚â§ F::NUM_BITS` (= 254 for
  bn254 `Fr`), so the 256-bit scalar must be split into multiple chunks, one per
  limb, with `max_bits = limb_bits`. `assign_privkey` does this (`NUM_LIMBS`
  limbs √ó `LIMB_BITS` bits).
- **Limb config must give `carry_mod` headroom.** With 4√ó64-bit limbs the
  `carry_mod` overflow bound (`n¬∑k ‚àí 1 + Fr::NUM_BITS ‚àí 2 = 507`) is below a
  256√ó256-bit product (512 bits) produced during scalar mult ‚Üí panic. halo2-ecc
  ships **88-bit √ó 3** limbs for secp256k1 (`configs/secp256k1/ecdsa_circuit.config`;
  bound 515 ‚â• 512); this crate uses the same.
- **Byte extraction is limb-config-independent.** `field_point_to_le_bytes`
  reads the low 256 bits of the positional truncation (range-checked to 256, so
  bits ‚â• 256 are constrained to 0), so the chip config can be chosen for
  soundness/efficiency without affecting the bridge.
- **Native Keccak in the test, in-circuit Keccak in Phase 3.** Phase 2 proves the
  bytes are the correct preimage (native keccak check); the in-circuit Keccak
  hash is Phase 3, after the Keccak gadget is ported to the axiom eDSL. Address
  byte order: `keccak256(x_be‚Äñy_be)` ‚Äî reverse each 32-byte half to big-endian
  before hashing.

## 11. Phase 3 ‚Äî axiom gadget ports + address bridge (in progress, 2026-07-04)

All three remaining gadgets are ported to the axiom `Context` eDSL, and the
migration's crux ‚Äî `keccak(pubkey)‚Üíaddress` ‚Äî is proven end-to-end in one
axiom circuit. The only remaining Phase 3 work is the `ZKMistV2Claim` rewrite
wiring everything together (plus the binding logic + 4 negatives).

### 11.1 Keccak port ‚Äî `circuits/src/keccak_axiom.rs`

Bit-level Keccak-f[1600] on halo2-base: 1600 boolean cells; Œ∏/œá/Œπ as polynomial
identities on bits (`XOR=a+b‚àí2ab`, `AND=ab`, `ANDNOT=(1‚àía)¬∑b`); œÅ/œÄ free
reindexing. Single-block ‚â§135-byte sponge (ZKMist's 64-byte pubkey only).
Verified in `tests/keccak_axiom.rs` (empty vector, 64-byte vs `tiny_keccak`,
and the privkey=1‚Üíaddress ground truth). Cost ~1.7M advice cells (algebraic œá).
A hand-copy typo in RC[17] (0x‚Ä¶0080 vs the correct 0x8000000000000080) silently
broke rounds 17-23 and was caught by the round-limited differential helper
`keccak_f1600_rounds`; the gadget's RC is now byte-identical to `keccak.rs`.

### 11.2 Merkle + nullifier ports ‚Äî `merkle_axiom.rs`, `nullifier_axiom.rs`

Thin wrappers over `poseidon_axiom` (halo2-base sponge). Merkle verifies a
26-level path via `gate.select` routing + `hash_interior`; nullifier is
`poseidon(key, domain)`. Both have self-consistency tests (circuit == native
halo2-base convention). They inherit the **sponge-convention caveat** (¬ß9.1):
the off-chain `zkmist-merkle-tree` uses light-poseidon, so the committed root /
nullifier must be rebuilt under halo2-base's convention (or the hasher wrapped)
before these verify a deployed tree.

### 11.3 Address bridge (the crux) ‚Äî `tests/address_bridge_axiom.rs`

Proves `address = keccak256(privkey¬∑G)[12..]` in ONE axiom circuit, wiring
`secp_axiom` (pubkey ‚Üí 64 LE bytes) ‚Üí reverse to BE ‚Üí `keccak_axiom` ‚Üí address.
Each of the 20 address bytes is constrained equal to the native Ethereum
derivation; MockProver (k=21, ~1.86M advice cells) asserts every constraint.
The derived address matches the Phase-2 secp byte-bridge test (`0xbc6d‚Ä¶a466`).
This is the capability halo2wrong could not provide and the reason the axiom
migration was chosen ‚Äî now proven end-to-end in-circuit.

### 11.5 Claim circuit + full soundness ‚Äî `circuits/src/claim_axiom.rs`

`prove_claim` wires every gadget into one `Context`-eDSL circuit proving a valid
claim: `privkey¬∑G ‚Üí keccak ‚Üí address ‚Üí poseidon(address)=leaf ‚Üí Merkle proof ‚Üí
root`, `nullifier = poseidon(key, domain)`, and a non-zero `uint160` recipient.
The ¬ß5/¬ß5a bindings: leaf‚Üîaddress (same cell), nullifier‚Üîscalar
(`recompose(limbs)==key_mod_p`), recipient‚Üîuint160 (range-check + non-zero),
and ‚Äî closing the ¬ß5a TRAP ‚Äî the explicit **`K < n_secp256k1` range proof**
(`secp_axiom::enforce_scalar_less_than_n`: limb range-checks + an MSB-first
limb-wise comparison) so a `K ‚â• n` forgery can't decouple `scalar¬∑G` from the
nullifier key.

Verified in `tests/claim_axiom.rs` ‚Äî the **happy path** plus the **four
forgery-rejection negatives** (wrong Merkle root, wrong nullifier, zero
recipient, `K ‚â• n`), each run full-circuit with `expect_satisfied(false)`.
The `K<n` proof is also covered in isolation by fast tests in
`tests/secp_axiom.rs` (valid key accepted, `K=n+1` rejected). MockProver k‚âà21
(~1.9M advice cells). The full axiom suite (21 tests across foundation /
poseidon / secp / keccak / address-bridge / merkle / nullifier / claim) is
green; the PSE stack is untouched.

## 12. Off-chain Merkle tree ‚Äî halo2-base convention (DONE 2026-07-04)

Resolved the ¬ß9.1 sponge-convention decision **adopt halo2-base end-to-end**:
nothing is deployed, so the off-chain eligibility tree is rebuilt under the
axiom circuit's convention rather than hand-rolling an unaudited Circom wrapper
around halo2-base's (non-public) raw permutation.

`merkle-tree/src/halo2base.rs` mirrors the crate's API under the halo2-base
sponge (capacity `2^64`, squeeze permutation, digest `state[1]`) ‚Äî the same

Verified two ways:
- **Byte-for-byte cross-check** (`circuits/tests/merkle_tree_halo2base.rs`):
  `merkle_tree::halo2base` `hash_leaf` / `hash_interior` equal
  `poseidon_axiom::native_hash_leaf` / `native_hash_interior`; plus an off-chain
  build ‚Üí proof ‚Üí verify round-trip (and a tampered-proof rejection).
- **End-to-end in-circuit** (`tests/claim_axiom.rs::
  test_axiom_claim_verifies_offchain_tree`): a tree built by the OFF-CHAIN
  tooling is verified by the IN-CIRCUIT claim logic ‚Äî the real production flow
  (eligibility list ‚Üí off-chain tree ‚Üí circuit proof), MockProver k‚âà21.

This closes the loop: the axiom claim circuit can now verify a real,
off-chain-built eligibility tree. (The streaming builder for the 64M-address

## 13. Real-KZG round-trip (DONE 2026-07-04) ‚Äî production blocker #2 cleared

under **real KZG** (SHPLONK multi-open + Blake2b transcript)? ‚Äî is answered:
**yes.** Using halo2-base's `bench_builder` (gen_srs ‚Üí keygen_vk ‚Üí keygen_pk ‚Üí

| stage        | time    |
|--------------|---------|
| keygen_vk    | 32 s    |
| keygen_pk    | 21 s    |
| create_proof | 150 s   |
| verify       | 19 ms   |
| proof size   | 960 B   |

at k=21 (~1.9M advice cells), peaking well within the 28 GiB WSL2 box (k=23's
~25 GiB OOM is a 4√ó larger commitment grid). A small Poseidon round-trip
(`tests/axiom_real_kzg.rs`) validated the axiom KZG pipeline first.

This clears production blocker #2 (real-KZG ‚Üí verify) **at the circuit level**:
the axiom claim circuit produces a verifying real-KZG proof. **Public instances

productionization:
- **Port the CLI prover** (`cli/src/halo2_prover.rs`, PSE k=23) to the axiom
