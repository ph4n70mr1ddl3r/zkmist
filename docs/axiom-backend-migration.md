# Axiom backend migration (halo2-ecc + halo2-base) ŌĆö production-ready on a 32 GB machine

> **Goal:** migrate ZKMist's entire circuit stack from the PSE halo2 backend to
> the **axiom** halo2 stack (`halo2-axiom` + `halo2curves-axiom` + `halo2-base`
> + `halo2-ecc`). This is the one path that delivers all three production wins
> at once: **k=18 (~1 GiB proving ŌĆö fits any machine, including 32 GB and
> claimants' laptops)**, an **audited secp256k1** (halo2-ecc), and a **wirable
> `keccak(pubkey)ŌåÆaddress` bridge** (halo2-ecc exposes pubkey bytes ŌĆö the bridge
> halo2wrong's CRT limbs cannot do).
>
> Originates from the 2026-07-03 investigation: both PSE-compatible options are
> blocked (halo2wrong: no byte extraction; halo2-ecc-as-chip-swap: axiom curve
> conflict). The conflict is only avoided by migrating the **whole** backend.

## 1. Why this is worth a multi-week rewrite

| Path | Proving RAM | secp audit | Keccak bridge | Claimant UX |
|---|---|---|---|---|
| Audit hand-rolled secp (stay PSE, k=23) | ~25 GiB | via audit | works (byte-aligned limbs) | poor (heavy prove) |
| **Axiom migration (k=18)** | **~1 GiB** | **audited lib** | **works** | **great** |

For an airdrop targeting ~1M ordinary claimants, ~25 GiB `zkmist prove` excludes
most users; ~1 GiB lets anyone prove on a laptop. That tips the effort toward
worth-it.

## 2. The axiom stack (replaces PSE)

| `halo2_proofs` (PSE fork v0.3.0) | `halo2-axiom` (= `halo2_proofs_axiom`) | proving backend |

### 9.1 Sponge-convention divergence ŌĆö Phase 3 reconciliation required

The PERMUTATION matches light-poseidon/Circom exactly (verified above), but the
HASH functions differ by construction:

| Aspect           | halo2-base (this gadget)               | light-poseidon / Circom (`zkmist-merkle-tree`) |
|------------------|----------------------------------------|------------------------------------------------|
| Capacity element | `2^64`                                 | `0`                                            |
| Input padding    | `1` then `0`s                          | none for a full RATE block                     |
| Squeeze          | extra empty permutation when `len % RATE == 0` | single permutation                      |
| Digest output    | `state[1]`                             | `state[0]`                                     |

The off-chain Merkle tree (`zkmist-merkle-tree`) and the existing PSE circuit
commit to the **light-poseidon** convention. Phase 3 must reconcile this so the
circuit verifies the *same* tree that is committed on-chain ŌĆö either:

- **(a)** rebuild the off-chain Merkle tree + nullifier under halo2-base's
  convention (audited, standard sponge ŌĆö preferred), or
- **(b)** wrap the axiom chip to replicate the Circom convention (capacity 0,
  no squeeze, output `state[0]`) so existing tree tooling keeps working.

This does not affect Phase 1/2 ŌĆö the chip and its parameters are correct either
way. It is a wiring decision for the full-circuit rewrite (Phase 3).

## 10. Phase 2 ŌĆö secp256k1 `scalar┬ĘG` + pubkey byte-bridge (DONE 2026-07-04)

Proved, in an isolated circuit, the capability halo2wrong could NOT provide and
that motivated this whole migration: deriving the secp256k1 **public key bytes**
from `privkey ┬Ę G` so they can feed `keccak256 ŌåÆ Ethereum address`. Deliverables:

- **`circuits/src/secp_axiom.rs`** ŌĆö axiom-stack secp256k1 gadget exposing:
  - `pubkey_from_privkey` ŌĆö `privkey ┬Ę G` via halo2-ecc's audited
    `EccChip::fixed_base_scalar_mult`.
  - `field_point_to_le_bytes` ŌĆö the **byte-bridge**: extract an secp256k1-Fp
    coordinate as 32 constrained little-endian bytes, ready for the Keccak
    preimage. (halo2-ecc's `ProperCrtUint` keeps a **positional** truncation
    alongside its CRT native cell ŌĆö unlike halo2wrong's pure CRT residues ŌĆö so
    each limb decomposes cleanly into bits/bytes.)
- **`circuits/tests/secp_axiom.rs`** ŌĆö two tests:
  1. `test_halo2ecc_secp256k1_pubkey_byte_bridge` ŌĆö in-circuit `scalar┬ĘG` is
     constrained equal to the native pubkey (`FpChip::assert_equal`); the pubkey
     coordinates are extracted as 32 LE bytes each; the 64 bytes recompose to the
     coordinates exactly AND `keccak256(x_beŌĆ¢y_be)[12..]` equals the Ethereum
     address. MockProver (k=18, ~131k advice cells) asserts every constraint.
  2. `test_eth_address_known_vector_privkey_one` ŌĆö anchors the native
     `privkey┬ĘG ŌåÆ keccak(pubkey)` derivation to the canonical vector
     `privkey=1 ŌåÆ 0x7E5FŌĆ”5Bdf`, so the in-circuit-vs-native comparison is tied to
     Ethereum ground truth (not merely self-consistent).

### 10.1 Findings (don't re-discover)

- **Scalar chunking (handoff correction).** The handoff's `fixed_base_scalar_mult(
  ŌĆ”, 256, 4)` is wrong: halo2-ecc asserts `max_bits Ōēż F::NUM_BITS` (= 254 for
  bn254 `Fr`), so the 256-bit scalar must be split into multiple chunks, one per
  limb, with `max_bits = limb_bits`. `assign_privkey` does this (`NUM_LIMBS`
  limbs ├Ś `LIMB_BITS` bits).
- **Limb config must give `carry_mod` headroom.** With 4├Ś64-bit limbs the
  `carry_mod` overflow bound (`n┬Ęk ŌłÆ 1 + Fr::NUM_BITS ŌłÆ 2 = 507`) is below a
  256├Ś256-bit product (512 bits) produced during scalar mult ŌåÆ panic. halo2-ecc
  ships **88-bit ├Ś 3** limbs for secp256k1 (`configs/secp256k1/ecdsa_circuit.config`;
  bound 515 Ōēź 512); this crate uses the same.
- **Byte extraction is limb-config-independent.** `field_point_to_le_bytes`
  reads the low 256 bits of the positional truncation (range-checked to 256, so
  bits Ōēź 256 are constrained to 0), so the chip config can be chosen for
  soundness/efficiency without affecting the bridge.
- **Native Keccak in the test, in-circuit Keccak in Phase 3.** Phase 2 proves the
  bytes are the correct preimage (native keccak check); the in-circuit Keccak
  hash is Phase 3, after the Keccak gadget is ported to the axiom eDSL. Address
  byte order: `keccak256(x_beŌĆ¢y_be)` ŌĆö reverse each 32-byte half to big-endian
  before hashing.

### 11.2 Merkle + nullifier ports ŌĆö `merkle_axiom.rs`, `nullifier_axiom.rs`

### 11.3 Address bridge (the crux) ŌĆö `tests/address_bridge_axiom.rs`

## 15b. Keccak kŌēł21 ŌåÆ k=18 ŌĆö analysis (optional optimization, 2026-07-04)

**Goal:** shrink the axiom circuit from kŌēł21 (~1.9M advice cells; Keccak alone
Ōēł1.69M) to k=18 (~262k rows, ~1 GiB proving) via a lookup-table Žć.

**Finding ŌĆö Žć-lookup alone is insufficient, and the API is blocked:**
- The Keccak-f[1600] cost is split across ╬Ė (column parity + D-XOR into all 25
  lanes) and Žć (the non-linear ANDNOT/XOR). Measured ~1.69M advice cells; k=18
  needs Ōēł250k, an ~85% cut. Eliminating Žć alone (the lookup target) leaves ╬Ė+╬╣,
  which still lands around kŌēł20 ŌĆö **k=18 needs BOTH ╬Ė and Žć lookup-based**
  (the design PSE `zkevm-circuits` uses).
- **halo2-base's public API exposes no custom lookup tables.** `LookupAnyManager`
  is a single, fixed range table (0..2^lookup_bits); there is no public way to
  add a second table for a 5-bitŌåÆ5-bit Žć (or a ╬Ė parity table) without forking
  `RangeCircuitBuilder`/`RangeConfig`. So a lookup-Žć is not an incremental change
  on top of `keccak_axiom.rs` ŌĆö it requires either (a) extending halo2-base with
  a custom-lookup API + rewriting Keccak ╬Ė+Žć over it, or (b) adopting PSE
  `zkevm-circuits`' axiom-stack Keccak (which ships lookup-based ╬Ė+Žć).

**Done now (minor):** removed a redundant `range_check(byte, 8)` before
`num_to_bits` in `keccak256` (the bit decomposition already forces byte < 256);
saves the input-byte lookups (no effect on the no-input-byte cell count; small
win for the 64-byte pubkey case).

**Recommendation:** k=18 is a real but sizable follow-up ŌĆö adopt PSE
`zkevm-circuits`' axiom Keccak (preferred: audited, lookup-based ╬Ė+Žć) or extend
halo2-base's lookup API. Until then the axiom circuit is kŌēł21 (~6ŌĆō10 GiB
proving), already a large win over the PSE k=23 (~25 GiB).

## 16. PSE stack removal ŌĆö blocker (2026-07-04)

The legacy PSE stack removal is **blocked** on the off-chain Merkle-tree builder
convention. Findings from attempting it:

- The axiom claim circuit uses the **halo2-base** Poseidon convention; the CLI's
  production `prove`/`bench`/`fetch` tree-building (`build_tree_streaming`,
  `ark_poseidon_hasher`) is **light-poseidon (Circom)**. So `zkmist prove`
  (axiom, the default) currently feeds a *light-poseidon* witness to a
  *halo2-base* circuit ŌĆö the proof would be rejected (root mismatch). This is a
  latent bug the PSE removal exposes. (The fixture path ŌĆö `gen-roundtrip-fixture`
  ŌĆö was fixed to halo2-base; the production `prove` flow was not.)
- `merkle-tree/src/halo2base.rs` only has an **in-memory** builder
  (`build_tree_with_depth`); there is **no streaming** halo2-base builder for the
  ~64M-address production tree. `build_tree_streaming` / `build_single_leaf_proof`
  (light-poseidon) have no halo2-base streaming equivalent.
- The circuits PSE removal is **clean and tested in isolation**, but can't land
  alone: `cli/src/halo2_prover.rs` (PSE) references the deleted circuits modules
  (`ZKMistV2Claim`, `poseidon`, `nullifier`, ŌĆ”), so the CLI removal is forced at
  the same time ŌĆö and that's where the tree-builder blocker sits.

**Unblock (prerequisite):** port `build_tree_streaming_with_depth` (+ the cache /
manifest logic) to the halo2-base convention in `merkle-tree`, then switch the
CLI's `prove`/`bench`/`gen-roundtrip-fixture` tree-building to it. Then the
coordinated removal (circuits + CLI + contracts PSE verifier) can land.

This is the same item as the deployment-gap "eligibility Merkle tree (halo2-base
convention)" ŌĆö the production tree-build path. Until then, the PSE stack stays as
`--pse` opt-in and the axiom `prove` flow is dev/fixture-only.
